CODEX MASTER BUILD SPECIFICATION
Custom GPT Industrial Control-Panel Drawing Workflow
Version: 2.0
Revision basis: sample-project review + controlled grading/self-improvement design
Encoding: UTF-8

======================================================================
0. PURPOSE OF THIS FILE
======================================================================

This is the sole bootstrap specification for a new Codex project. It is
self-contained and does not depend on previous chat history.

PRIMARY PRODUCT GOAL

For each eligible project, generate these three primary PDF deliverables:

  01_生管課用圖_<工程編號>.pdf
  02_鈑金施工圖_<工程編號>.pdf
  03_沖孔施工圖_<工程編號>.pdf

`生管課用圖` and `生管用圖` are recognized as aliases for the same output
class. The canonical project-facing label is `生管課用圖` unless an approved
company naming rule requires otherwise.

The generator may use only explicitly allowlisted current-project Excel
workbooks/worksheets, explicit current user corrections, frozen production
Knowledge, approved deterministic rules, and approved reusable component or
CAD-library metadata. It must not use data from:

- any file or worksheet classified as `生管文件`;
- any file or worksheet classified as `電機施工圖`;
- completed or modified target drawings for 生管課用圖/生管用圖,
  鈑金施工圖, or 沖孔施工圖;
- extracted text, thumbnails, filenames, hashes, reviewer findings, or other
  derivatives of those forbidden references.

The intended workflow is:

  allowed Excel documents
  -> file and worksheet adjudication
  -> positive generator allowlist and contamination scan
  -> normalized job_spec.json with cell-level provenance
  -> conflict/TBD handling and schema validation
  -> one canonical drawing_model.json
  -> three deterministic PDF renderers
  -> optional editable DXF source artifacts
  -> cross-PDF consistency and fabrication checks
  -> freeze and hash generated output
  -> sheet-level reference manifest and adjudication
  -> blind deterministic grading and independent review
  -> paired portfolio evaluation
  -> proposal-first, regression-tested optimization
  -> protected held-out evaluation
  -> private ChatGPT web Preview with mandatory human CAD review

The system must reduce long-context degradation by treating chat as temporary
working memory and repository files as durable, versioned memory.

This file contains:

- project scope, privacy, safety, and source-boundary rules;
- repository architecture and canonical artifact contracts;
- parent-thread and subagent orchestration;
- context-limit and fresh-thread recovery rules;
- PDF-generation, validation, comparison, and grading requirements;
- controlled self-improvement techniques adapted from modern agent systems;
- every implementation phase and completion gate;
- reusable prompts, including a rebuilt initial Codex prompt;
- resource references.

The root Codex thread is the COORDINATOR. It executes this file phase by
phase, explicitly delegates bounded work, waits for and validates every child
result, writes durable checkpoints, and stops only at explicit human gates or
genuine blockers.

Do not paste old chat logs into the project. Do not use remembered chat
content as authority. This file, accepted repository artifacts, approved
source manifests, and signed human decisions are the authority.

2. NON-NEGOTIABLE PROJECT RULES
======================================================================

2.1 SOURCE IMMUTABILITY

- Original archives, project folders, catalogs, drawings, spreadsheets, and
  completed work are immutable.
- Temporary extraction is allowed only under workspace/tmp/ or a versioned
  evaluation sandbox.
- Temporary files must be reproducible and disposable.
- Raw customer files, contracts, forbidden references, and completed drawings
  must not be committed to Git.
- A source manifest must record hashes before parsing or copying.

2.2 PRIMARY OUTPUT CONTRACT

The required product outputs are three PDFs:

- 生管課用圖 PDF;
- 鈑金施工圖 PDF;
- 沖孔施工圖 PDF.

Editable DXF files may be produced as secondary artifacts when the approved
renderer supports them. A DXF does not substitute for a required PDF. Each PDF
must be readable, associated with the correct project, and internally
consistent with the other two PDFs.

2.3 GENERATOR INPUT BOUNDARY — POSITIVE ALLOWLIST ONLY

The generator receives a newly constructed bundle. It must not browse the
original project tree.

Allowed project-specific evidence:

1. Current-project Excel workbooks explicitly approved by the source guard.
2. Only the approved worksheets within those workbooks.
3. Explicit current user corrections recorded in a signed decision artifact.
4. Approved deterministic rules with versioned rule IDs.

Allowed reusable evidence:

5. Frozen production Knowledge.
6. Approved component/catalog metadata and CAD block metadata.
7. Company conventions explicitly marked as defaults, never hidden facts.

Always forbidden to the generator, regardless of folder or filename:

- `生管文件` and any copy, export, cached text, or embedded worksheet derived
  from it;
- `電機施工圖` and any copy, image, PDF, CAD export, or extracted text derived
  from it;
- completed target drawings, modified target drawings, historical target
  thumbnails, OCR, vectors, hashes, page counts, filenames, and reviewer
  annotations;
- evaluation expected answers, reference-adjudication records, held-out
  answers, score reports, or comparison overlays;
- symlinks, junctions, environment variables, search indexes, or alternate
  paths that reveal forbidden data.

Folder-name filtering alone is insufficient. Build an explicit file-and-sheet
allowlist, then scan content hashes, archive members, symlinks/junctions,
filenames, extracted strings, and reference-only sentinels before generation.

Any forbidden exposure is `REFERENCE_LEAKAGE` and invalidates the run.

2.4 STALE WORKSHEET AND TEMPLATE REJECTION

Workbook selection is an engineering gate. Legacy `.xls` and `.xlsx` files may
contain old examples or unrelated customer tabs.

For every candidate worksheet, record and compare:

- engineering/project number;
- customer name;
- project name;
- approved revision/date marker when present;
- identifiers shared with other allowed current-project workbooks;
- whether formulas or named ranges point to stale/hidden sheets.

A worksheet whose identifiers conflict with the current project is
`STALE_TEMPLATE_SHEET` and is excluded. A sheet with insufficient identity is
`HUMAN_REVIEW_REQUIRED` and cannot provide critical fabrication values.
Using a stale or unrelated sheet is a hard evaluation failure.

Hidden sheets, very hidden sheets, external links, cached values, named ranges,
print areas, and formulas must be inventoried. The system must not assume the
active tab is current.

2.5 NO UNSUPPORTED FABRICATION VALUES

Never invent or silently infer the following without an explicit, traceable
allowed source or approved deterministic rule:

- voltage, current, SCCR, enclosure/IP/NEMA rating;
- material, sheet thickness, surface finish, wire size;
- exact enclosure, panel, plate, door, rail, duct, or cutout dimensions;
- exact hole/cutout location, bend allowance, bend deduction, or spacing;
- model number, manufacturer, quantity, rating, or protection class;
- drawing approval, certification, safety, or fabrication-release status.

Missing or conflicting values must use one of:

- TBD
- UNVERIFIED
- CONFLICT
- HUMAN_REVIEW_REQUIRED

A proposal may be shown only when labelled `PROPOSED`, linked to its rule or
analogy, and excluded from verified-fact scoring. A draft must never be called
certified, approved, safe for production, or ready for fabrication without
explicit human sign-off.

2.6 FIELD-SPECIFIC SOURCE AUTHORITY WITHIN THE ALLOWED BUNDLE

Do not assume one workbook type is universally more authoritative than another.
Create a versioned `source_authority_profile.json` that maps each field family
such as project identity, enclosure dimensions, material, device quantity,
commercial description, or revision status to its approved source precedence.

Default handling:

1. Explicit current user correction recorded for this project.
2. Field-specific authority rule for the current revision.
3. Accepted canonical project decision.
4. Approved deterministic derivation rule.
5. Frozen reusable catalog/CAD metadata for catalog facts only.
6. Company default only when explicitly authorized and labelled as a default.

When two allowed sources of equal or unclear authority disagree, record
`CONFLICT`; do not guess based on filename, modified time, or historical target.
Do not use completed target drawings to resolve a generator-side conflict. They
are reviewer evidence only after output freeze.

2.7 DEVELOPMENT MATERIAL IS NOT PRODUCTION KNOWLEDGE

Keep separate:

- deployable Custom GPT Instructions;
- production Knowledge files;
- repository skills used by Codex;
- development/reference casebooks;
- evaluation cases and expected behavior;
- failed generated outputs;
- completed historical outputs;
- reviewer reports and grading artifacts;
- optimization notes and rejected candidates;
- protected held-out answers.

Do not upload failed outputs, reviewer commentary, project-specific completed
answers, reference manifests, or held-out data as Custom GPT Knowledge.

2.8 COMPLETED WORK IS ADJUDICATED EVIDENCE, NOT PERFECT GROUND TRUTH

Every generated/reference mismatch receives one class:

- EXPLICIT_IN_ALLOWED_INPUT
- DERIVABLE_BY_APPROVED_RULE
- CONFLICTING_ALLOWED_INPUT
- DESIGN_CHOICE_WITH_CONSTRAINTS
- UNAVAILABLE_IN_ALLOWED_INPUT
- OPTIONAL_COMPANY_CONVENTION
- REFERENCE_POSSIBLY_INCORRECT
- UNRESOLVED

Do not penalize a generated TBD/CONFLICT when the completed drawing contains a
value unavailable or contradictory in the allowed input bundle. Do penalize an
unsupported invented value. Exact historical placement is not mandatory when
the source only defines constraints and the generated alternative is valid.

2.9 ONE CANONICAL MODEL, THREE RENDERERS

The generator must create one `drawing_model_<工程編號>.json` and render all
three PDFs from it. The three renderers may emphasize different information,
but they must share panel IDs, dimensions, materials, quantities, orientations,
and common geometry.

Independent free-form generation of three unrelated drawings is prohibited.
Cross-PDF inconsistency is a hard gate or score cap depending on severity.

2.10 SHEET-LEVEL REVISION CONTROL

A `修改` file may replace one sheet rather than the whole drawing package.
References and generated revisions must be tracked by project, output type,
panel/sheet identity, revision, effective status, and `supersedes` links.
“Newest filename wins” is prohibited.

2.11 PRIVACY AND EXTERNAL PROCESSING

- Default execution is local-only.
- Do not transmit project data to an API until explicit approval is recorded
  in docs/PRIVACY_APPROVAL.md.
- Never store credentials in the repository.
- Use environment variables for credentials.
- Record every external file transmission, model/provider, retention context,
  and cleanup action.
- Keep the final Custom GPT private during initial testing.

2.12 FROZEN RELEASES AND FROZEN GRADERS

Once a candidate release is frozen, evaluation-harness work may not alter its
Instructions, Knowledge, skills, schemas, source classifier, extraction logic,
drawing model, renderers, validators, grading profile, or tolerance profile.
Any improvement creates a new versioned candidate.

The grading profile and reference adjudication are also frozen per experiment.
Do not move the goalposts after seeing candidate output.

2.13 PROPOSAL-FIRST SELF-IMPROVEMENT

Agents may propose changes but may not directly mutate accepted Instructions,
production Knowledge, active skills, shared renderers, or the accepted grading
profile. Every proposal must include:

- target artifact and current hash;
- evidence and affected failure IDs;
- exact diff;
- expected benefit;
- regression risks;
- regression test added first;
- rollback information;
- independent review status;
- human approval when required.

Applying a stale proposal whose target hash changed is prohibited.

2.14 SAMPLE-DERIVED HAZARDS — EVALUATION-ONLY KNOWLEDGE

The reviewed sample archive established hazards that must become tests, not
hidden generator hints:

- target-like PDFs may appear beneath an `電機施工圖` area, so path exclusion is
  insufficient;
- legacy workbooks may contain stale unrelated tabs;
- allowed workbooks can disagree on dimensions or materials;
- completed references may include exact decisions absent from allowed Excel;
- revision PDFs may replace only selected panels/sheets;
- most historical PDFs may be raster scans, limiting vector comparison;
- project size can range from one panel to many panels, so portfolio scoring
  must not be dominated by one large project.

Specific project IDs, completed layouts, and hidden decisions remain outside
production Knowledge and outside every generator bundle.

3. LONG-CONTEXT AND THREAD MANAGEMENT MODEL
======================================================================

3.1 WHAT CODEX CAN AND CANNOT DO

The root Codex app conversation can explicitly spawn subagent threads,
wait for them, steer them, inspect them, and close them. Those are child
agent threads within the current task.

The root conversation must not claim it can autonomously create a new
independent top-level Codex app conversation. When a top-level context
reset is required, it must create durable checkpoint files and ask the
human to open a fresh Codex conversation using the continuation prompt
in section 18.

For isolated executable mock-agent runs, the project may launch fresh
`codex exec --ephemeral` child processes. These are separate execution
sessions and are preferred for blind generator/reviewer isolation.

3.2 DURABLE MEMORY

The conversation is temporary. Durable memory lives in:

- AGENTS.md
- docs/00_PROJECT_CHARTER.md
- docs/01_CURRENT_STATE.md
- docs/02_ARCHITECTURE.md
- docs/03_KNOWLEDGE_MANIFEST.md
- docs/04_DECISION_LEDGER.md
- docs/05_EVAL_SUITE.md
- docs/06_EVAL_LEDGER.md
- docs/07_CHANGELOG.md
- docs/SOURCE_ROOTS.md
- docs/SESSION_CHECKPOINT.md
- orchestration/TASK_REGISTRY.csv
- orchestration/NEXT_THREAD_PROMPT.txt
- release manifests and Git commits.

Do not carry state by copying an old chat transcript into a new thread.

3.3 PROACTIVE RESET RULE

The coordinator must checkpoint:

- after every major phase;
- after every accepted or rejected optimization cycle;
- before and after a six-project fan-out;
- after at most three optimization cycles in one root conversation;
- whenever competing drafts, stale assumptions, or large raw outputs are
  accumulating;
- before the conversation approaches a practical context limit.

Do not wait for hard truncation.

At a reset:

1. Finish or safely stop active child agents.
2. Validate and close child threads.
3. Update every canonical state file.
4. Write docs/SESSION_CHECKPOINT.md.
5. Write orchestration/NEXT_THREAD_PROMPT.txt.
6. Commit the checkpoint.
7. Return the status NEW_TOP_LEVEL_THREAD_REQUIRED.
8. Tell the human to start a fresh Codex app conversation and paste only
   the short continuation prompt from section 18.

3.4 KEEP THE COORDINATOR CONTEXT SMALL

- Subagents write detailed results to files.
- The coordinator receives compact, structured summaries.
- Never paste complete PDFs, CAD dumps, huge CSVs, or raw logs into the
  parent response.
- Refer to paths, hashes, finding IDs, and schema-valid summaries.
- The coordinator reads only the minimum files required for the current
  phase.
- Historical detail belongs in ledgers and Git, not CURRENT_STATE.

3.5 RUN TRAJECTORIES AS A FLIGHT RECORDER

Every generator, reviewer, grader, optimizer, and implementation run writes a
redacted machine-readable trajectory containing:

- run/task ID and parent ID;
- requested and actual model;
- prompt/instruction/skill/config hashes;
- visible-file manifest and allowlist hash;
- tool calls, commands, timestamps, exit codes, and timeouts;
- active sandbox and network policy;
- input and output artifact hashes;
- validator results and termination reason;
- compaction/context-reset events when observable.

Trajectories may contain sensitive paths or prompts and must be protected like
evaluation evidence. They support root-cause analysis but are never uploaded to
production Knowledge.

3.6 MEMORY TIERS

Use three explicit tiers:

- ACCEPTED_DURABLE: approved rules, schemas, decisions, releases, and tests;
- PROJECT_EVIDENCE: immutable manifests, run artifacts, references, scores;
- WORKING_NOTES: temporary hypotheses that expire or are archived.

No agent may promote working notes to accepted durable memory without the
proposal and review process.

4. PARENT COORDINATOR AND SUBAGENT ORCHESTRATION
======================================================================

4.1 GLOBAL AGENT CONFIGURATION

Create `.codex/config.toml` with:

[agents]
max_threads = 6
max_depth = 1
job_max_runtime_seconds = 1800

Keep depth at 1 unless an independently reviewed reason requires otherwise.
The coordinator explicitly requests subagents; it does not assume automatic
fan-out.

4.2 PROJECT-SCOPED CUSTOM AGENTS

Create narrow TOML agent definitions under `.codex/agents/`. Each definition
must include `name`, `description`, and `developer_instructions`. Omitted model
or sandbox settings inherit from the parent, so every run records the effective
values.

1. repo_explorer
   - read-only;
   - inventories repository/source architecture;
   - returns evidence and paths, not edits.

2. source_guard
   - read-only over immutable sources and writable only to its designated
     preparation/manifests directory;
   - classifies files and individual worksheets;
   - builds the positive generator allowlist;
   - detects forbidden content, hidden sheets, stale templates, external links,
     duplicates, aliases, and leakage sentinels;
   - cannot generate drawings or view reviewer conclusions.

3. implementation_worker
   - workspace-write only within an approved bounded task/worktree;
   - implements one accepted proposal;
   - cannot modify unrelated shared artifacts.

4. knowledge_curator
   - read-only on source/evaluation data;
   - writes only to a candidate Knowledge directory;
   - separates reusable rules from project-specific answers;
   - cannot apply candidate Knowledge to an accepted release.

5. case_generator
   - workspace-write only inside one isolated generator workspace;
   - sees only the allowlisted generator bundle;
   - creates job_spec, drawing_model, three PDFs, optional DXF, provenance,
     validation report, and completion record;
   - cannot access references or prior reviewer output.

6. deterministic_grader
   - runs schema, PDF, geometry, BOM, cross-output, leakage, and consistency
     checks using frozen code and profiles;
   - does not change generated or reference artifacts;
   - reports parser/alignment limitations rather than false passes.

7. reference_adjudicator
   - read-only over references and writable only to its designated
     adjudication directory;
   - starts only after generated outputs are frozen;
   - builds the effective sheet set and supersedes graph;
   - classifies reference elements by source availability;
   - cannot edit the generator release or generated output.

8. project_reviewer
   - read-only over frozen generated outputs, allowed inputs, adjudicated
     references, and deterministic evidence;
   - produces schema-valid findings with source classification and root cause;
   - cannot implement recommendations.

9. blind_ab_reviewer
   - read-only;
   - sees anonymized A/B outputs and allowed project inputs;
   - reports A preferred, B preferred, equivalent, or neither acceptable;
   - is not told which output is baseline or candidate.

10. portfolio_optimizer
    - read-only;
    - reads structured project results, trajectories, and aggregate metrics;
    - clusters failures by root cause and proposes the smallest general change;
    - cannot access protected held-out answers or implement changes.

11. skill_curator
    - activates only after sufficient scored runs exist;
    - reviews skill duplication, trigger quality, size, conflicts, and usage;
    - creates hash-bound proposals under optimization/skill_proposals/;
    - cannot write active `.agents/skills/*/SKILL.md` directly.

12. independent_auditor
    - read-only by default;
    - checks requirements, hashes, tests, score calculations, Git diff,
      experiment integrity, and human gates;
    - may propose fixes but does not silently implement them.

4.4 SUBAGENT LAUNCH PROTOCOL

Before spawning any agent:

1. Create `orchestration/tasks/<task_id>.md`.
2. Define exact inputs, outputs, writable paths, forbidden paths, completion
   criteria, timeout, and output schema.
3. Add the task to TASK_REGISTRY.csv.
4. Create an isolated worktree/output directory where writes could conflict.
5. Generate and hash the visible-file manifest.
6. Confirm whether the task may access allowed inputs, generated output,
   references, or held-out data.

Then explicitly instruct Codex to spawn the named agent, wait for it, route
clarifications only to that agent, inspect its result, and close it.

For repeated project tasks, use `spawn_agents_on_csv` only when supported by
the installed Codex version and after a synthetic test. Otherwise spawn
explicit children. Never exceed six concurrent child threads.

4.5 PARENT MONITORING DUTIES

The coordinator must:

- record every child ID and effective configuration;
- monitor status, heartbeat, timeout, and approval requests;
- wait for all required children before aggregation;
- validate expected files, schemas, hashes, and trajectory completion;
- rerun failure with a new task/run ID rather than overwriting evidence;
- reject partial, malformed, or prose-only results;
- compare declared file changes against Git diff and artifact hashes;
- close completed child threads;
- merge/apply child work serially;
- remain the only writer to accepted shared workflow/configuration files.

4.6 NO CONCURRENT SHARED WRITES

Parallel agents write only inside assigned workspaces. The coordinator alone
may write or apply:

- accepted Instructions and production Knowledge;
- active `.agents/skills`;
- shared schemas, renderers, validators, and grading profiles;
- canonical state files, backlog, releases, and cohort assignments.

4.7 REQUIRED AGENT INTERACTION FLOW

Use this data flow:

  source_guard
      -> generator_input_manifest + approved worksheet allowlist
      -> case_generator
      -> deterministic validation
      -> generation freeze + hashes

  reference_adjudicator (starts after freeze)
      -> effective reference sheet manifest + adjudication
      -> deterministic_grader
      -> project_reviewer
      -> grading_result + comparison report

  multiple project results
      -> portfolio_optimizer
      -> proposal package
      -> regression test first
      -> implementation_worker under coordinator control
      -> blind paired rerun
      -> independent_auditor
      -> accept/reject/human gate

No backward edge may expose completed references, scores, or reviewer findings
to a generator run being graded.

6.13 TOP-LEVEL GRADING RESULT

Every run reports:

Validity: PASS | FAIL
Quality score: 0.0–100.0
Scorable coverage: 0–100%
Confidence: HIGH | MEDIUM | LOW
Critical findings: integer
High findings: integer

A quality score without coverage and confidence is invalid.

6.14 HARD VALIDITY GATES

A run is FAIL when any occur:

1. Reference leakage or forbidden file/sheet exposure.
2. Use of a stale or unrelated worksheet.
3. Missing required PDF without a valid machine-readable failure record.
4. Blank, unreadable, corrupted, or wrong-project PDF.
5. Invented critical value or unsupported verified claim.
6. Silent resolution of critical source conflict.
7. Missing provenance for a critical generated value.
8. Post-freeze artifact mutation.
9. Critical contradiction among the three PDFs.
10. Unsupported approval/certification/fabrication-readiness claim.

Recommended non-invalidating score caps:

- complete panel/sheet missing: maximum 50;
- wrong explicit enclosure dimension or material: maximum 70;
- high-severity constructability defect: maximum 75;
- missing one required output-specific information class: maximum 80.

6.15 100-POINT RUBRIC

Use these weights:

- Source selection, provenance, and conflict handling: 20
- Panel schedule and enclosure facts: 15
- Device/BOM/quantity/tag fidelity: 20
- Shared geometry and cross-PDF consistency: 15
- 生管課用圖-specific quality: 10
- 沖孔施工圖-specific quality: 10
- 鈑金施工圖-specific quality: 10

Total: 100.

Each scorable item receives:

- 1.00 correct/within tolerance;
- 0.75 minor non-fabrication discrepancy;
- 0.50 partially correct or acceptable alternative incompletely shown;
- 0.25 recognizable intent but major correction required;
- 0.00 missing, wrong, or unusable.

Criticality multipliers:

- 5 fabrication/safety/identity critical;
- 3 engineering-significant;
- 1 drafting/style preference.

Dimension score = dimension weight multiplied by the weighted item-score ratio.
An invented critical value triggers a hard gate instead of merely scoring zero.

6.16 SCORABLE COVERAGE AND CONFIDENCE

Scorable coverage is the weighted proportion of relevant reference items that
are explicit, derivable, or governed by approved constraints.

- HIGH: coverage >=85%, reliable pairing/registration, no unresolved critical
  reference items;
- MEDIUM: coverage 65–84% or moderate pairing/reference uncertainty;
- LOW: coverage <65% or substantial unresolved/unavailable reference content.

Low coverage does not prove poor output. It means the comparison cannot support
a strong optimization conclusion.

6.17 BOM AND OBJECT COMPARISON

Use weighted precision, recall, and F1 for device/BOM comparison so missing and
invented items are both penalized. Score repeated identical devices as quantity
groups where practical.

Match geometry by stable object identity and minimum-cost assignment using
class, panel region, tag, position, size, rotation, and relationships. Preferred
comparison order:

1. structured drawing model;
2. vector/CAD objects;
3. registered edge/shape comparison;
4. raster image diagnostics;
5. blind expert review.

Raw pixel similarity, SSIM, or perceptual similarity may diagnose rendering and
overall structure, but optional visual-convention items are capped at five
points across the entire project. Pixel similarity cannot override engineering
facts or constraints.

6.18 TOLERANCE PROFILES

Use versioned tolerances by object family and measurement type. Never apply one
universal millimetre tolerance to enclosures, cutouts, devices, labels, rails,
ducts, and title text. Every tolerance records authority, soft and hard limits,
units, scope, version, and human approval status.

6.19 PROJECT-LEVEL AGGREGATION

Evaluation hierarchy:

Portfolio -> Project -> Panel/logical fabrication unit -> Objects/constraints.

Compute detailed object scores, aggregate panels to one project score, then
average projects equally unless a predeclared risk weighting is frozen before
the experiment. A project with many terminals or many panels must not dominate
the portfolio merely because it has more objects.

Repeated runs within one project are not independent projects.

6.20 PAIRED CANDIDATE COMPARISON

Baseline and candidate use the same project bundles, grader/profile, reference
adjudication, model policy, and repeated-run count. Report:

- candidate minus baseline by project and dimension;
- mean and median paired difference;
- win/tie/loss rate;
- worst-project regression;
- run-to-run standard deviation;
- project-level bootstrap confidence interval when enough projects exist;
- new/resolved critical and high findings.

6.21 INITIAL ACCEPTANCE THRESHOLDS

Provisional acceptance requires all of:

- all hard gates pass;
- overall paired improvement >= +2 points;
- targeted dimension improvement >= +5 points;
- no project regression worse than -3 points;
- no new critical or high-severity defect;
- no material increase in run-to-run variability;
- new regression test passes;
- independent evidence or blind A/B review supports the change.

These thresholds must later be calibrated against reviewer repeatability.

6.22 BLIND A/B REVIEW

Use blind A/B review for at least 20% of development comparisons and every
held-out project. Ask which requires less correction, is safer to send to CAD
review, is clearer for fabrication, and has fewer technically significant
errors. Allowed outcomes: A preferred, B preferred, equivalent, neither
acceptable. Numeric scoring remains primary; A/B is independent confirmation.

6.23 REVIEWER CALIBRATION

At least 10% of development projects, every disputed critical finding, and a
sample of held-out projects must be scored independently by two qualified
reviewers. Report mean absolute score difference, category agreement, and
weighted kappa or intraclass correlation when sample size supports it.

Do not tighten acceptance thresholds below normal reviewer noise. Disagreement
that changes an accept/reject decision requires adjudication and a rubric or
tolerance clarification before the next optimization cycle.

6.24 PRODUCTION UTILITY — SEPARATE DASHBOARD

Do not merge these directly into the historical 100-point score at first:

- CAD correction minutes;
- added/deleted/moved/resized/relabelled object count;
- critical edit count;
- review cycles and clarification questions;
- percentage retaining generated main layout;
- shop questions and drawing-caused rework.

An optimization that raises historical similarity but increases engineering
correction effort is not a genuine improvement.

6.25 TRAJECTORY CONTRACT

Every run trajectory is schema-valid, redacted, immutable after completion,
and linked by hash from its run manifest. It is used to diagnose whether a
failure originated in source selection, extraction, prompt/skill activation,
model reasoning, rendering, or grading.

6.26 SKILL PROPOSAL CONTRACT

A skill proposal is stored as `PROPOSAL.md` plus metadata, not as an active
`SKILL.md`. It binds to the current target hash and includes scanner/test state,
support files, expected trigger, non-trigger examples, rollback metadata, and
review decision. Only the coordinator may apply an approved proposal to
`.agents/skills`.

======================================================================
16. PHASE 9 — CONTROLLED, PROPOSAL-FIRST OPTIMIZATION LOOP
======================================================================

Goal: implement evidence-backed, versioned improvements without prompt bloat,
reference leakage, autonomous mutation, or overfitting.

Run no more than three optimization cycles in one top-level conversation.
Checkpoint and request a fresh top-level thread afterward.

16.1 ONE CONCEPTUAL CHANGE PER CYCLE

For each cycle:

1. Select the highest-priority unresolved root cause.
2. Verify evidence across projects and trajectories.
3. Decide the correct layer: source classifier, worksheet selection, extraction,
   schema, skill, drawing model, renderer, validator, report, or rubric.
4. Reject insufficiently supported recommendations.
5. Add a regression test before changing behavior.
6. Create a hash-bound proposal with exact diff and rollback.
7. Create a new candidate release/worktree.
8. Make one conceptual change whenever practical.
9. Run unit/schema/synthetic tests.
10. Rerun the same development projects and all regression cases.
11. Keep model policy, grader, tolerances, references, and run count fixed.
12. Conduct blind A/B review where required.
13. Compute paired project-level deltas and uncertainty.
14. Accept or reject using section 6.21 and all hard gates.
15. Preserve positive and negative evidence; commit separately.

16.2 AVOID PROMPT BLOAT

Before changing Instructions:

- check whether the behavior is already covered;
- test whether the true defect is source selection, extraction, schema,
  renderer, validation, or skill activation;
- identify overlap/conflict with existing rules;
- define the regression evidence;
- remove or merge superseded language;
- reject instruction growth without measurable benefit.

16.3 TRAJECTORY-BASED DIAGNOSIS

Use trajectories to answer:

- Did the correct skill trigger?
- Did the generator see only the approved sheets?
- Was the correct value extracted but lost in drawing_model?
- Did the renderer omit or alter a correct model element?
- Did the grader mis-pair or misclassify a reference?

Do not edit the prompt merely because final PDFs differ.

16.4 ACCEPT/REJECT

Reject on any new leakage, unsupported critical value, stale-sheet use, invalid
schema/model, missing PDF, unreadable PDF/DXF, lost provenance, critical score
regression, project drop >3 points, increased high-severity findings, or failure
to reproduce improvement independently.

Accept only when all hard gates pass, targeted and overall thresholds pass,
variance does not materially worsen, regression tests pass, independent/blind
evidence supports the change, and the release is reproducible and hashed.

16.5 SKILL CURATION

After approximately 20–30 trustworthy scored runs, the skill_curator may propose:

- merging overlapping skills;
- improving descriptions/triggers;
- adding non-trigger boundaries;
- archiving unused or harmful skills;
- splitting oversized skills;
- promoting a repeated, approved procedure into a reusable skill.

A lesson becomes a durable company rule only when it appears in at least three
independent projects or prevents one critical failure and receives engineering
approval. One unusual historical layout normally remains a project-specific
exception.

16.6 EVOLUTIONARY CANDIDATE SEARCH — EXPERIMENTAL ONLY

Techniques similar to evolutionary skill/prompt optimization may be tested only
after the evaluator is stable. Limit them to one narrow skill or instruction
module at a time. Generate multiple candidate variants, evaluate with frozen
development/regression sets, apply constraint and size gates, and route the best
variant through the same proposal, audit, and human-approval process.

Never permit an evolutionary loop to:

- edit the accepted release directly;
- access held-out answers;
- change the grader during candidate search;
- optimize solely for keyword/pixel similarity or a single model judge;
- mutate source-isolation or safety rules;
- apply a candidate without exact diff, tests, and rollback.

16.7 CYCLE ARTIFACTS

Each cycle creates:

optimization/cycles/cycle-N/
  hypothesis.md
  evidence.json
  proposal_manifest.json
  proposed_change.md
  exact_diff.patch
  regression_test.md
  before_scores.csv
  after_scores.csv
  paired_comparison.json
  blind_ab_review.json when required
  decision.md
  rollback.md
  codex_handoff.md

Stop early when critical domain clarification is required, evidence/coverage is
insufficient, reviewers materially disagree, tolerance is undefined, source
information is unavailable, no candidate passes regression, or
READY_FOR_HELD_OUT_GATE is reached.

======================================================================
17. PHASE 10 — HELD-OUT GATE, PRIVATE WEB PREVIEW, AND CONTINUOUS LEARNING
======================================================================

17.1 HELD-OUT PROTECTION

A project is held-out eligible only when required allowed inputs and effective
completed outputs are verified, it is absent from production Knowledge and the
development reference casebook, and its completed output has not been opened for
detailed diagnosis or candidate optimization.

If a held-out case is opened for detailed diagnosis, permanently move it to
regression and replace it.

17.2 FINAL HELD-OUT RUN

Use at least three held-out projects and three fresh independent runs per
project. Do not alter the candidate, grader, tolerance profile, or reference
adjudication during the gate.

Non-negotiable gates:

- zero reference leakage;
- zero stale-sheet use;
- zero fabricated critical values;
- zero unsupported approval/certification claims;
- 100% schema-valid job_spec and drawing_model outputs;
- all three PDFs produced or explicitly failed for a valid reason;
- every critical value source-supported or marked TBD/CONFLICT/UNVERIFIED/
  HUMAN_REVIEW_REQUIRED;
- all artifacts open and all hashes/metadata complete;
- no critical cross-PDF inconsistency.

Provisional quality gates:

- project-equal held-out average >=90/100;
- each project average >=85/100;
- no individual run below 80/100;
- run-to-run standard deviation <=5 points per project;
- no project regression worse than 3 points;
- no unresolved high-severity mismatch derivable from allowed inputs;
- portfolio coverage/confidence sufficient to support the conclusion;
- blind A/B review does not favor the accepted baseline on material quality.

Allowed status:

- READY_FOR_PRIVATE_PREVIEW
- NOT_READY
- INSUFFICIENT_EVIDENCE

Never output PRODUCTION_APPROVED.

17.3 CHATGPT WEB PREVIEW

The automated harness cannot replace testing in the actual private Custom GPT.
Create or duplicate a private candidate using direct Configuration view. Paste
approved Instructions, upload only the approved Knowledge manifest, enable Code
Interpreter & Data Analysis, and keep other tools disabled unless explicitly
designed.

Run at least:

1. One ordinary complete project.
2. One complex multi-sheet project.
3. One incomplete/conflicting-input project.

Verify file/worksheet recognition, forbidden-source rejection, Knowledge/skill
behavior, PDF downloads, Chinese names/text, fresh-chat consistency, drawing
model creation, three-PDF consistency, and TBD/conflict behavior.

Initial launch status is PRIVATE PILOT only, with mandatory human CAD review and
no automatic production/fabrication release.

17.4 INGESTING EACH NEW PROJECT

Assign each incoming project a feature vector before cohort selection, including
panel count, output page count, enclosure type, device-family diversity, source
conflict count, stale-sheet risk, revision structure, missing-data rate, raster
versus vector reference, and similarity to existing cases. Use these tags to
maintain a coverage matrix and avoid filling development or held-out cohorts
with near-duplicates.

For every new completed project:

1. Register immutable allowed and forbidden source manifests.
2. Run blind generation before opening completed targets.
3. Freeze and hash the generated package.
4. Build effective sheet-level reference manifest.
5. Adjudicate explicit, derivable, conflicting, unavailable, constrained,
   optional, or suspect reference elements.
6. Grade and review with the frozen profile.
7. Add objective failures as regression cases before changing behavior.
8. Cluster root cause with existing evidence.
9. Propose the smallest generalizable change.
10. Rerun development and regression cohorts.
11. Preserve untouched projects for future held-out evaluation.
12. Record production-utility outcomes when human correction is available.

17.5 PRODUCTION UTILITY FEEDBACK

When an engineer edits a generated draft, record correction time, object edits,
critical corrections, review cycles, clarification questions, accepted main
layout, shop questions, and drawing-caused rework. Keep this as a separate
longitudinal dashboard until enough data supports calibrated weighting.

19. RESEARCH-BACKED DESIGN PATTERNS AND RESOURCE REFERENCES
======================================================================

19.1 CODEX PATTERNS TO USE

- Explicit bounded subagents with clear return formats; the coordinator waits,
  inspects, and closes them.
- `codex exec` for isolated non-interactive generator/reviewer runs using only
  flags supported by the installed CLI.
- Repository-scoped `.agents/skills` with progressive disclosure for reusable,
  narrow workflows.
- Project `AGENTS.md` for stable routing rules, not as a giant duplicated spec.
- Project-scoped `.codex/config.toml` and versioned agent definitions.
- Traces/trajectories plus evals to diagnose and improve agent workflows.

19.2 HERMES-INSPIRED PATTERNS WORTH IMPLEMENTING

- Learn from execution traces, but turn lessons into candidate skills or rules,
  never immediate live mutation.
- Curate, merge, and prune skills after sufficient evidence accumulates.
- Test evolutionary candidate generation only after the grader is stable and
  only on narrow modules with deterministic gates.
- Keep accepted durable memory separate from raw session history.

Hermes-style self-evolution remains experimental for this project. Its useful
idea is candidate generation plus evaluation; autonomous application is not
accepted.

19.3 OPENCLAW-INSPIRED PATTERNS WORTH IMPLEMENTING

- Proposal-first skill updates: candidate content is not live until explicitly
  applied.
- Bind proposals to current hashes so stale proposals cannot clobber changes.
- Record rollback metadata and scanner/test state.
- Capture redacted per-run trajectories as a flight recorder.
- Use workspace and agent isolation plus tool/skill allowlists.
- Build small deterministic benchmark scenarios before complex model judges.

19.4 PATTERNS TO AVOID

- Autonomous changes to the live Custom GPT.
- Automatic learning from every historical drawing.
- A generator grading itself.
- Hidden shared memory between generator and reviewer.
- Pixel similarity or a model judge as the primary reward.
- Optimization on held-out completed drawings.
- Community skills or plugins operating on private customer data without source
  review, pinning, sandboxing, and approval.
- Fine-tuning or reinforcement learning before evaluator repeatability and data
  governance are established.

======================================================================
20. FINAL SUCCESS DEFINITION
======================================================================

The project succeeds when it produces a private-pilot Custom GPT package and a
reproducible evaluation/self-improvement system that:

- accepts only approved current-project Excel worksheets as project evidence;
- excludes `生管文件`, `電機施工圖`, and all completed-target derivatives;
- rejects stale or unrelated workbook sheets;
- extracts source-supported facts with cell-level provenance;
- detects conflicts and refuses to invent fabrication values;
- creates schema-valid job_spec and one canonical drawing model;
- generates readable 生管課用圖, 鈑金施工圖, and 沖孔施工圖 PDFs;
- keeps all three PDFs internally consistent;
- creates useful indexes, provenance, validation, and trace reports;
- reconstructs reference revisions at sheet level;
- grades validity, quality, scorable coverage, and confidence;
- separates source-unavailable human decisions from genuine failures;
- compares baseline and candidate as paired project-level experiments;
- improves through proposal-first, versioned, regression-tested changes;
- uses trajectories to locate the true root-cause layer;
- preserves protected held-out projects;
- tracks real human correction effort as production utility;
- remains stable across fresh runs;
- requires mandatory human CAD/fabrication review.

No automated component may declare PRODUCTION_APPROVED. The highest automated
status is READY_FOR_PRIVATE_PREVIEW. The first real launch is PRIVATE PILOT.
